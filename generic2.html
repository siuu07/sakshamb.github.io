UFV ALUMNI WEBSITE

In the winter of 2021, I had worked with a team of 5 members to create a website that would allow for Alumni-Student interactions through numerous fun activities. It was a project that went on for 3 months and was divided into 11 phases starting from coming up with a model for the back-end and then implementing a GUI accordingly. This project helped me build a strong foundation in data manipulation techniques using SQL queries and debugging for redundancies.

The project was divided into the following phases-

1.PLANNING

Here, we defined the mission statement of our database. The primary aim of our database is to act as a mediator amongst the University of the Fraser Valley’s alumni. In the planning phase we came up with the following points-

  1. Users must be able to update/delete their respective comments as they wish.
  2. Visitors can see the homepage, however, to see the comments one must be registered into the website.
  3. Only the admin user must have the capability of proofreading and allowing comments to be added.
  4. The website must be light in weight and not overloaded with graphics.

a) Physical Data Model

We came up with a physical data model in the first phase of the Website creation process to identify the relationships between each data table. The physical data model aided in visualizing database structure by reproducing database column keys, constraints, indexes, triggers, and other RDBMS features. In the above table data types used for the Table Final depend on the tables Users and Display as it is a view table which is depended on the base table to get its data.

b) Data Flow Diagram

Although the data flow diagram was completed in the later stages of the website design, we had initially visualised by identifying how the information would flow throughout the website to allow for a more organised appraoch.

2.SYSTEM DEFINITION

Here, we gave permission to the different people that access the database.

    Administrators-

  1. Can edit the different webpages.
  2. Can read all added comments.
  3. Can enable comments.
  4. Can see all users.
  5. Can delete a user profile

    Website Visitors-

  1. Can see the homepage.
  2. Are not allowed to add comments until registered.

    Website Users-

  1. Can add a comment.
  2. Can view all comments that are added.
  3. Can update/delete their own comment.
  4. Can edit or delete their own profile.

3.REQUIREMENTS COLLECTION AND ANALYSIS

In this phase, we first made of the requirements that are needed for our project (Fraser Valley India Alumni website) and then we collected some records from the previous alumni students of the Fraser valley India (FVI). Also, one thing we analysed that a lot of previous alumni students are present on various social platforms such as Instagram, so we created a database that can enable the users to enter their Instagram ID.

4.DATABASE DESIGN

We kept the primary objectives in our mind, and we completed the conceptual, physical, and logical database design, which are explained briefly in the Data Flow Diagram (DFD) & Enterprise Relationships Diagram (ERD) sections.

5.DBMS SELECTION

As our group had decided to create a dynamic website. We decided to use php as it would provide easy connection to with oracle database and is the best combination with html and CSS So with the help of php we connected to Apache server oracle database (sqldeveloper) and used XAMPP for server-side connection.

6. APPLICATION DESIGN

The architecture of the user interface and the application systems that use and process the database is referred to as application design. Since the GUI is one of the most critical elements of the framework, we made certain that it was easy to read, use, and understand. Application architecture is split into two parts: transaction design and user interface design.

Transaction Design

When the user wants to update his/her comment or his/her profile transaction design is performed.

User Interface Design

Before introducing forms and reports, we ensured that they had descriptive titles and that the instructions were short, specific, and easy to understand for the user. It had all of the appropriate fields grouped in a sequential order. In the forms/reports, no abbreviations were used. There was also a consistent use of colour, making it physically pleasing to the consumer.

7. PROTOTYPING

Prototyping is a rough model but partially working model of a database system. For the testing purposes of our project, we added various aliases so as in to check the comments are updated successfully or not, and if the details of the user are being updated (username, password, email IDs) etc. We also asked some friends to sign up for the website and put in their comments which helped us to understand the project ever more. The feedback from the friends helped us to change various things in our website (such as font style, addition of videos and images) etc.

8. IMPLEMENTATION

Initially, we created two tables User consisting of the following fields- name, username, password. Another table "comment" was created containing the user's name, username, password and comment . We joined these two tables to create a display table however, this created a Redundancy issue due to the common fields in both the tables. Then we did not request the username etc. in the comments table instead we used a PHP session to acquire the stored username inside the display table. After this, we created a final (view) table where we added the username, name and email from the users table and we added batch, field, Instagram, comment from the display table by equating the primary key (username) in both tables.We created a back-end database in this phase, the local DBMS using (SQL developer and OCI Oracle cloud infrastructure) and then a suitable connection was made to the front-end design using NetBeans, PHP, HTML, CSS, BOOTSTRAP, Java Script, jQuery). The following languages were used to design and code the framework of the website. We had then used BOOTSTRAP and PHP forms at the front-end of the website to insert/fetch/update the data from the users into the website. After the designing of the website and back-end database was done, we used XAMPP and Apache Server to connect the front-end and back-end of our website. After that, we were able to run our Fraser Valley India (FVI) website on local host.

9. DATA CONVERSION & LOADING

Data Conversion is the translation of computer data from one format to another. Throughout a computer environment, data is encoded in a variety of ways. Whenever any one of these variables is changed, data must be converted in some way before it can be used by a different computer, operating system, or program.

    In our case-

  1. The data in the Private table is being transported into the Display table. Hence, we have converted all these records into the format required by the Display table.
  2. When we input the password however, while adding this into the database we convert this into an encrypted form.

10. TESTING

  • We tested our website mainly to check if all the different webpages were free of errors. After this, we checked whether an admin could perform all the primary functions like adding/deleting users and had access to the comments added by them.
  • We checked if while registering or logging in the user enters any details that are incorrect or inadequate the appropriate message is displayed.
  • We tested if all comments were first reaching the Private table and then being added to the Final table by the admin.

11. MAINTENANCE ISSUES

  1. Problems with connection: After the front-end (using Neatbeans, PHP, HTML, CSS, BOOTSTRAP, Java Script, Jquery) and back-end (SQl, OCI Oracle Cloud Infrastructure) of our Fraser Valley India (FVI) website, was completed, we faced a major issue where we weren’t able to connect out front-end interface with back-end interface. As a result, the users were just able to view website, but were not able to put in any information as it was prompting an error. After rigorous research by all the team members, we were finally able to join front-end and back-end using XAMPP and Apache server. XAMPP is a server-side connection which basically provides port numbers and connection (https – 443) to website which can be opened by a local host.
  2. Database records couldn’t be updated: In between designing out website, it was pointed out by one of our teammates that, whenever a user was trying to update their account details, they were getting an error for the same. As a result, the records of the database were not getting updated. After, careful proof reading of the codes, it was found out that one of the queries were wrongly written and implemented. So after carefully examining, the variables and function in the queries were changed properly and then, the users were able to update their account details.

RESULTS

One of the main challenges we faced initially was understanding how complex of a project we should aim for. It was hard to come up with something that each member would agree on. Finally, we decided to go for a project that would relate to what we were doing as it would motivate us to keep going. Hence, we came up with a way for a student to interact with their alumni through the use of comments and other features. An instrumental part of the project that stood out to me in terms of its significance was documenting each phase thoroughly. It became extremely simple to map out our progress by taking down every little change we made. Another essential skill that I developed through the responsibilities assigned to me was Normalization and debugging for redundancies in the data.
